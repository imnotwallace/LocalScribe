# Stage 5.2 smoke runbook - Session Details window + two-column speaker & identity management

GUI-only acceptance (cannot be run headless). Run on the Win11 box against a real build and record
results inline at the bottom. Fix any failure before finishing the branch.

Run: `dotnet run --project src/LocalScribe.App`

## Prerequisites

- Build is green: `dotnet build LocalScribe.slnx` (0 warnings / 0 errors); `dotnet test tests/LocalScribe.App.Tests`
  (175/175). The 2 `LocalScribe.Core.Tests` fixture failures (`Der_within_baseline_plus_epsilon`,
  `Golden_pair_wer_stays_at_baseline`) are pre-existing env-absent DER/golden fixtures - not Stage 5.2 regressions.
- At least one **finalized** session (for the details window + speaker checks). Two finalized sessions help F1/F8
  (place-and-reopen, dedup-per-id).
- A **Settings identity** set (Settings -> your name + role) BEFORE F3, so `SessionBootstrap` mints the self
  participant that renders in the Local column as "you". If the identity is empty, no self participant is minted and
  F3's "you" line is expected to be ABSENT (note that, don't fail F3).
- A **Matter with a roster member**, and a session **tagged to that matter**, for F2/F4/F6 (create the matter, add a
  roster person, then tag a session to it in that session's details window Classification card).
- For F6 (Split speakers), the diariser prerequisite from the Stage 5 runbook applies: `LocalScribe.Diarizer.exe`
  must be published beside the app (self-contained single-file with `-p:IncludeNativeLibrariesForSelfExtract=true`) or
  Run fails with `HELPER_CRASH`. See `docs/plans/2026-07-04-stage-5-smoke-runbook.md` "Prerequisite" for the exact
  publish + copy-only-the-.exe steps. F6 needs a splittable session with a Remote leg tagged to a matter whose roster
  (or the session's own remote participants) includes at least one named remote person (e.g. "Barrister").

## F1 - Open the Session Details window from Sessions ("Open detail"); auto-save; dedup-per-id

Steps: on the Sessions page, right-click a session row -> "Open detail" (first context-menu item, above "Open").
Edit a field (e.g. change the Title, or add a Description) and click away (LostFocus). Then, with the window still
open, right-click the SAME session row -> "Open detail" again.
Expected: the first "Open detail" opens a dedicated "Session details" window (Mica title bar, its own taskbar icon)
populated from that session. Editing a field auto-saves (a "Saved" indicator appears briefly; no Save button). The
SECOND "Open detail" on the same session ACTIVATES the already-open window (brings it to front) rather than opening a
duplicate - there is only ever one details window per session id.
Record: pass/fail + confirm auto-save indicator fired + confirm no duplicate window.

## F2 - Reuse the SAME window from Matters (by id)

Steps: go to the Matters page, expand the matter that has a tagged session, and click that tagged session's "Open".
Expected: it opens the SAME "Session details" window (not the read view) for that session id. If the details window
for that id is already open (e.g. from F1), Matters' "Open" activates the existing one - same dedup-by-id path as
Sessions. (The transcript READ view remains reachable separately from the Sessions page via double-click or the
"Open" context-menu item - confirm that still works and is distinct from "Open detail".)
Record: pass/fail + confirm Matters "Open" -> details window (not read view) + read view still reachable from Sessions.

## F3 - Two-column Local/Remote speaker lists render; per-side free-text add; self shows as "you"

Steps: with a Settings identity set (see Prerequisites), open a session's details window. Inspect the Speakers
section. In the LOCAL column's "Add local speaker" box type a name and click Add; in the REMOTE column's "Add remote
speaker" box type a name and click Add.
Expected: participants render as TWO ordered columns (Local | Remote). The self participant appears in the LOCAL
column with a muted "(you)" label. The local add lands in the Local column; the remote add lands in the Remote
column. Each row has a Remove control that drops that participant from its side. (If no Settings identity is set, the
"(you)" line is legitimately absent - note it.)
Record: pass/fail + self "(you)" present/absent (and whether identity was set) + local/remote adds land on the
correct side.

## F4 - Per-side roster add (fixes the old "everything is remote" bug)

Steps: in a details window for a session tagged to a matter with a roster, use the ROSTER picker + "Add from roster"
button in the REMOTE column to add a roster member. Then remove it and add a (different or same) roster member via
the LOCAL column's roster add.
Expected: the roster member added via the REMOTE column lands in the Remote list; the one added via the LOCAL column
lands in the Local list. This is the fix for the prior bug where roster-add hard-coded Remote regardless of intent.
Record: pass/fail + confirm remote-column roster add -> Remote AND local-column roster add -> Local.

## F5 - "Speaker counts follow these lists" toggle (auto-set from data on open; override; count==1 tier-2)

Steps: (a) On a session where the declared counts already match the number of named participants per side, open the
details window and observe the "Speaker counts follow these lists" toggle and the Local/Remote count boxes. Add a
speaker to a side and watch the count. (b) Toggle "follow" OFF and confirm the count boxes become editable; type a
manual count. (c) System-mix case: on a session whose declared Remote count EXCEEDS its named remote participants
(e.g. RemoteCount = 3 with 1 named remote - a Webex leg with 3 people, 1 identified), open the details window and
observe the toggle state, then add a remote name and re-check RemoteCount. (d) With a session that has exactly ONE
named local speaker (LocalCount 1), open that session's read view and confirm its local lines are labelled with that
speaker's name.
Expected: (a) toggle is ON (data matches lists); count boxes are read-only/disabled and track the list sizes as you
add/remove. (b) toggling OFF makes the counts manually editable and a typed value sticks (not re-derived). (c) the
system-mix session opens with the toggle OFF (auto-set from data, per the design decision) so the DECLARED RemoteCount
= 3 is PRESERVED - adding a remote name does NOT silently lower it to the named count; RemoteCount stays 3 until you
explicitly tick the box. (d) the single-named local speaker's name labels the local lines in the read view
(count==1 NameResolver tier-2).
Record: pass/fail for a/b/c/d + for (c) confirm the declared count survived the edit.

## F6 - Split-speakers cluster naming uses a participant picker (B2)

Steps: open Split speakers on a splittable session (Remote leg, tagged to a matter / with named remote participants
- see Prerequisites). Run a Remote source. In the naming step, click a Remote cluster's name field.
Expected: the name field is an editable DROPDOWN listing that side's participants (e.g. "Barrister") - AND still
accepts free text for an un-rostered speaker. Pick "Barrister" for a Remote cluster, Confirm, then open that session's
read view: those cluster's lines now read "Barrister". (Local clusters offer the LOCAL participants, not the remote
ones.)
Record: pass/fail + confirm the dropdown lists the right side's participants + confirm the picked name persists to the
read view (writes via `speakers.Names`).

## F7 - Deleting a session closes its open details window (WindowRegistry)

Steps: open a session's details window. Leave it open. Go to the Sessions page and Delete that same session (context
menu -> "Delete (to Recycle Bin)...", confirm Yes).
Expected: the open details window for that session CLOSES automatically (via `WindowRegistry.CloseAllFor`) as part of
the delete, BEFORE the session folder is recycled - no orphaned window pointing at a deleted session, no locked-file
failure recycling the folder. The per-window editor is disposed on that close (no leak).
Record: pass/fail + confirm the window closed on delete + confirm the folder recycled cleanly.

## F8 - Capture-exclusion + window placement persist for the details window

Steps: (a) In Settings, ensure "Exclude app windows from screen capture" (Privacy) is ON; open a details window and
verify it is excluded from capture (e.g. it does not appear in a screen-share preview / screen recording). Toggle the
Privacy setting while the window is open and confirm the exclusion re-applies live. (b) Move and resize the details
window, close it, and reopen a details window - confirm it restores to roughly the same size/position (state key
`sessionDetailsDefault`; clamped on-screen).
Expected: (a) the window honors the capture-exclusion setting and re-applies it live when toggled. (b) size/position
persist across close/reopen and stay on-screen. (Note: two DIFFERENT sessions' detail windows share one saved
placement, so opening a second session's window over the first is expected - not a bug.)
Record: pass/fail for a/b + any capture-exclusion or placement anomaly.
