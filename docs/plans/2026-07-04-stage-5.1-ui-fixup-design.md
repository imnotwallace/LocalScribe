# Stage 5.1 Design: UI Fixup — Theming, Chrome, Layout, Sessions IA

Date: 2026-07-04
Status: Approved (brainstorm dialogue 2026-07-04). Refines the Stage 4 manager
(docs/plans/2026-07-03-stage-4-manager-design.md) and Stage 3b tray/overlay
(docs/plans/2026-07-02-stage-3b-tray-liveview-overlay.md). No schema changes; no
behavioural change to capture/transcription/persistence. Pure UI/UX remediation.

## 1. Scope

Stage 5.1 fixes the visual and interaction defects observed on a real Win11
desktop launch of the Stage 4/5 manager. The app currently reads as broken: a dark
Fluent shell with a white content island, unreadable colour clashes, clipped
controls, a window with no caption buttons, and a blank/absent tray icon. This
stage makes the app look and behave like a finished, native Windows tray app.

It delivers seven workstreams:

- **WS1 — Theming foundation.** Make WPF-UI theming authoritative and
  system-following (light/dark tracks Windows).
- **WS2 — Session list re-theme.** Replace the unthemed `ListView`/`GridView`
  with a Fluent-themed `DataGrid`; purge hardcoded ARGB brushes.
- **WS3 — Window chrome, tray, and branding.** Add a `ui:TitleBar` so the window
  is closable/minimisable; ship a branded multi-resolution `.ico` for
  exe/taskbar/window/tray.
- **WS4 — Responsive layout.** No control clips at any supported window size;
  toolbars wrap/scroll; the Details pane grows with the window.
- **WS5 — Sessions Details IA redesign.** Group the metadata editor into labelled
  sections, rebuild the participant editor, add button hierarchy, labels,
  placeholders, and empty states.
- **WS6 — Consent + Overlay theme consistency.** Remove the hardcoded light/blue
  escape-hatch palettes; consume Fluent brushes.
- **WS7 — Evidentiary safety.** Demote the inline per-row session **Delete** to
  the context menu with confirmation and danger styling.

### 1.1 User decisions (2026-07-04)

| Question | Decision |
|---|---|
| Overall theme direction | **Follow the system theme** (auto light/dark via `SystemThemeWatcher`). No hardcoded mode. |
| Details / metadata panel depth | **Redesign the information architecture** (grouped sections, rebuilt participant editor, button hierarchy) — not merely a contrast/sizing fix. |
| Session-list control | **Fluent-themed `DataGrid`** (sortable columns, native selection/hover) replacing `ListView`+`GridView`. |
| Minimise (`_`) behaviour | **Standard minimise-to-taskbar.** Only the **X** hides to tray (existing close-to-tray logic). |
| App/tray icon source | **Claude designs a branded multi-resolution `.ico`** (mic/scribe/waveform mark) shipped in this stage; user can swap later. |

### 1.2 Non-goals

- No changes to capture, transcription, diarisation, persistence, or schema.
- No new features (no search index, no export, no correction editing — all remain
  their own later stages).
- No rework of the read view / live view / split-speakers window layouts beyond the
  shared theming/brush cleanup they inherit from WS1/WS6.
- No change to the **close-to-tray-and-re-create** window lifecycle model (Stage 4
  decision): the X still tears the window down and the tray re-creates it. We only
  add the missing caption buttons and standard minimise.
- No transcript content deletion/redaction (evidentiary): WS7 makes accidental
  session-folder deletion *harder*, it does not add any content-removal capability.

## 2. Root-cause map (why the app looks broken today)

Established by a five-dimension code survey (theming, layout, tray/lifecycle,
pages/content, shell architecture). Each user-visible symptom maps to a concrete
cause and fix.

|---|---|---|---|

**Additional defects folded into scope:**

- Inline per-row **Delete** button sends the whole session folder to the Recycle
  Bin, styled like any benign control (`SessionsPage.xaml:116`) — evidentiary risk
  → WS7.
- Participant editor exposes four buttons, **two captioned "Add"**, right-docked so
  visual order reverses reading order (`SessionsPage.xaml:184-192`) → WS5.
- No primary/secondary/danger button hierarchy anywhere except ConsentDialog → WS5.
- Unlabeled inputs (search, matter filter, matter name, name/role) and cryptic

## 3. Design by workstream

### WS1 — Theming foundation (authoritative, system-following)

- `App.xaml`: change `ui:ThemesDictionary Theme="Dark"` to the neutral/default so it
  is not a compile-time lock. Keep `ControlsDictionary`.
- `App.OnStartup`: call `ApplicationThemeManager.Apply(...)` to stamp the resolved
  theme, then `SystemThemeWatcher.Watch(mainWindow)` so light/dark tracks the OS at
  runtime and on change. This is the missing standard startup path; it also removes
  the reason Mica/tray had to be defensively deferred (the ApplicationIdle deferral
  for `OpenMainWindow` stays — it is cheap insurance and independently documented).
- Because the theme becomes system-following, **every surface must resolve colours
  from Fluent `{DynamicResource}` brushes**, never literals — enforced by WS2/WS6.
- **Provide an app-level implicit `TextBlock` style** (WPF-UI ships none) whose
  `Foreground` binds `{DynamicResource TextFillColorPrimaryBrush}`, so any unstyled
  `TextBlock` is legible in both themes instead of defaulting to black. Add a
  secondary/muted text style for the existing `Foreground="Gray"` hints. This single
  change fixes the bulk of Issue 3 across every page.
- Constraint honoured: no new window may be shown synchronously before the message
  pump (the invisible-Mica gotcha, memory `wpfui-fluentwindow-startup-rendering`).

### WS2 — Session list re-theme (Fluent DataGrid)

- Replace the `ListView`+`GridView` in `SessionsPage.xaml` with a WPF-UI-styled
  `DataGrid` (or the Fluent `ui:DataGrid` if present in Wpf.Ui 4.0.3; otherwise a
  standard `DataGrid` with the Fluent theme applied). Columns: Title, App, Date
  (with tooltip), Duration, and the status-chips/actions cell.
- Preserve existing bindings (`Rows`, `SelectedRow`), double-click-to-open, and the
  context menu (Open / Reveal / Archive / Split speakers — plus Delete moving here
  per WS7). Keep virtualization.
- Delete the `#22808080` chip brush and any other hardcoded ARGB in the page; chips
  use a Fluent subtle-fill brush resource.
- Acceptance: the centre pane themes identically to the shell in both light and dark
  — no white island.

### WS3 — Window chrome, tray, and branding

- **Caption buttons:** add a `<ui:TitleBar>` to `MainWindow` (title + app icon +
  minimise/maximise/close). This restores all three caption buttons. `X` keeps the
  existing genuine-close (tears down; tray re-creates). `_` performs standard
  minimise-to-taskbar (user decision). Verify `TitleBar` sits correctly with the
  existing `DockPanel`/`InfoBar`/`NavigationView` layout.
- **Branded icon:** author a multi-resolution `.ico` (16/32/48/256 px) — a simple,
  professional LocalScribe mark (mic + waveform / scribe motif), monochrome-friendly
  so it reads at 16px. Store under `src/LocalScribe.App/Assets/` (new folder).
  - Wire `<ApplicationIcon>` in `LocalScribe.App.csproj` → exe/taskbar/Alt-Tab.
  - Set `MainWindow.Icon` (and the other FluentWindows) → title-bar/taskbar icon.
  - **Tray:** base the tray icon on the branded mark but **retain the
    idle/recording/paused colour semantics** as a tint/badge (the status signal is
    valuable during a call). If `GeneratedIconSource` timing remains fragile
    pre-pump, load the `.ico` as a real `Icon` resource so the tray never renders
    blank; apply the status colour as an overlay/tint rather than generating the
    whole glyph at runtime.
- Acceptance: tray icon is always visible and branded; window has working
  min/max/close; exe and taskbar show the LocalScribe icon.

### WS4 — Responsive layout

- **Toolbars** (Sessions + Matters): replace fixed-width horizontal `StackPanel`s
  with `WrapPanel` (or a toolbar that scrolls) so no control clips when the column
  is narrow. Give the search/filter controls sensible min widths but allow wrap.
- **Sessions Details column:** change from fixed `Width="380"` to a star/Auto column
  with a `MinWidth` (~340) and a sensible `MaxWidth` so it grows with the window
  instead of staying cramped. Keep the `GridSplitter`.
- **Scroll safety nets:** the Details `ScrollViewer` keeps vertical auto; ensure
  content wraps rather than requiring horizontal scroll (long names/labels wrap).
- **Window minimums:** raise `MainWindow.MinWidth` so the 3-pane layout (nav +
  list-min + splitter + details-min) never forces content off-screen; pick the value
  from the summed minimums (~940–980px) so nothing clips at the smallest size.
- Acceptance: at min size and at 1080×720, every control on every page is fully
  visible or reachable by an intended scrollbar; nothing is cut off at a column edge.

### WS5 — Sessions Details IA redesign

Adopt the Settings page's grouped-form *structure*, lifting a shared field/section
vocabulary into an app-level resource dictionary so Sessions, Matters, and Settings
share one design language. **Caveat:** Settings' `GroupBox` containers are not
actually Fluent-themed (native `GroupBox` renders a light etched border + black
header). The shared "section" container must therefore be a Fluent-themed surface
(e.g. `ui:Card` / `ui:CardExpander`, or a themed `GroupBox` style bound to Fluent
brushes) — not a raw `GroupBox` — and Settings' existing `GroupBox`es are migrated to
it too.

- **Sections** (each a `GroupBox`/card): **Session** (Title, Description),
  **Classification** (Medium — relabelled to plain language e.g. "Call type";
  Matters + show-archived), **Participants** (roster list + add flows),
  **Speakers** (Local / Remote counts), **Status** (Archived toggle).
- **Participant editor rebuild:** eliminate the two ambiguous "Add" buttons. Distinct,
  self-describing actions with primary/secondary styling and correct reading order
  (no order-reversing right-dock): e.g. "Add from roster" (pick + add), "Add name"
  (free text into this session), "Add to roster too" (free text into roster +
  session). Relabel "Roster target" to plain language (e.g. "Save new people to
  matter"). Group the add controls so their relationship is obvious.
- **Button hierarchy:** introduce shared `PrimaryButton` / `SecondaryButton` /
  `DangerButton` styles (Fluent accent for primary; subtle for secondary; danger
  colour for destructive). Apply across Sessions and Matters (Refresh, Add, Remove,
  Delete, Repair index, etc.).
- **Labels & placeholders:** every input gets a label or watermark — the search box
  ("Filter sessions…"), matter filter ("All matters"), matter name, name/role.
- **Empty states:** sessions grid shows "No sessions yet — press Record to start" /
  "No sessions match this filter"; the Details pane shows "Select a session to view
  its details" when nothing is selected (mirror MattersPage's `HasSelection`
  pattern) instead of a greyed wall of empty fields.
- Acceptance: a first-time lawyer can read every field/label, tell the buttons
  apart, and is never shown an undifferentiated wall of controls.

### WS6 — Consent + Overlay theme consistency

- `ConsentDialog`: remove `Style="{x:Null}"` and the hardcoded light-grey/blue/near-
  black literals; consume Fluent brushes so the dialog is legible in light and dark.
  The original defect it worked around (near-white text invisible on a plain light
  window) is resolved at the theme layer once WS1 makes theming authoritative — the
  dialog can stay a plain `Window` but must pull its colours from the merged Fluent
  resources (or become a `FluentWindow` if that is cleaner; either is acceptable as
  long as it themes with the app and does not reintroduce the pre-pump invisibility
  risk — the consent dialog runs a nested pump via `ShowDialog`, which warms
  composition, so a `FluentWindow` here is safe).
- `OverlayWindow`: replace the hardcoded `#E6202020` card + white text with Fluent
  brushes (retain deliberate translucency for the always-on-top overlay, but source
  the colours from resources so it reads in both themes).
- `SplitSpeakersWindow` / other pages: replace stray `#33..` accent literals with
  Fluent resources.
- Acceptance: no surface is a hardcoded light/dark island; grepping the XAML for
  literal `#` ARGB brushes returns only intentional, theme-agnostic values
  (e.g. overlay opacity), documented inline.

### WS7 — Evidentiary safety

- Remove the always-visible inline per-row **Delete** button from the session list
  cell. Keep delete available only via the **context menu** (where Archive / Split
  already live) and via the existing confirmation dialog (Stage 4: Recycle Bin +
  one confirm). Give the context-menu Delete danger styling and ensure the confirm
  clearly names the session.
- No change to the correction-only / non-destructive model; this only reduces
  accidental evidence loss (aligns with memory `no-transcript-content-deletion-evidentiary`).

## 4. Shared infrastructure introduced

- **`Assets/` folder** in `LocalScribe.App` for the `.ico` (and any future art).
- **App-level styles dictionary** (merged in `App.xaml`) hosting: button styles
  (`PrimaryButton`/`SecondaryButton`/`DangerButton`), the field/label/section
  vocabulary lifted from `SettingsPage`, and the chip/subtle-fill brush. This is the
  single design-language source Sessions/Matters/Settings all consume.

## 5. Testing strategy

This is a TDD, subagent-driven project (449 tests, 0 warnings gate). UI XAML is not
unit-testable end-to-end, so:

- **View-model / logic:** unchanged behaviour must keep all existing
  `LocalScribe.App.Tests` green (no VM API changes expected; WS5 relabels UI text and
  regroups controls but preserves bindings/commands). Any command/property rename
  gets its test updated in lockstep.
- **Theming / brush hygiene:** add a lightweight test (or build-time check) that the
  app-level resource dictionaries expose the named brushes/styles the pages reference,
  so a missing `{DynamicResource}` key fails fast rather than rendering wrong.
- **Static guard:** a test that greps shipped XAML for disallowed hardcoded ARGB
  brushes (allow-list the documented overlay-opacity exceptions) to prevent
  regression of the panda problem.
- **Manual smoke (GUI, real desktop):** the acceptance items below — cannot be
  headless (memory: 3b/4/5 all required GUI smoke).

## 6. Manual smoke checklist (Stage 5.1)

To be run on the Win11 box after implementation (GUI, not headless):

- E1. Launch on a **light** Windows theme → whole app is light and consistent
  (sidebar, list, details, dialogs). No white island, no unreadable text.
- E2. Switch Windows to **dark** while running → app follows to dark live.
- E3. Tray icon is present and branded; status colour reflects idle/recording/paused.
- E4. Window has working **minimise** (to taskbar), **maximise**, and **close**
  (X hides to tray; re-open from tray restores).
- E5. exe/taskbar/Alt-Tab show the LocalScribe icon.
- E6. Resize the window to its minimum and to 1080×720 → no control is clipped on
  Sessions, Matters, or Settings; toolbars wrap; Details pane grows.
- E7. Sessions Details pane reads as grouped sections; participant add buttons are
  each unambiguous; every input is labelled.
- E8. Empty states render (fresh/filtered-empty grid; no-selection details pane).
- E9. Session delete is reachable only from the context menu, danger-styled, with a
  naming confirmation; no inline row Delete button remains.
- E10. Consent dialog and overlay are legible in both light and dark.

## 7. Constraints & risks

- **Invisible Mica/FluentWindow before the pump** (memory
  `wpfui-fluentwindow-startup-rendering`): keep the ApplicationIdle deferral for the
  main window; do not introduce any new synchronously-shown FluentWindow at startup.
  Applying theming properly (WS1) is expected to make rendering more robust, not less.
- **Wpf.Ui 4.0.3 API surface:** confirm the exact `ApplicationThemeManager` /
  `SystemThemeWatcher` / `ui:TitleBar` / `ui:DataGrid` APIs against the installed
  version before coding (verify via the Wpf.Ui 4.0.3 assembly / official docs, not
  memory).
- **DataGrid binding parity:** the `GridView` → `DataGrid` swap must preserve every
  column, the chips/actions cell, selection, double-click, and context menu; this is
  the highest-churn change and gets its own careful review.
- **Icon quality at 16px:** the branded `.ico` must remain legible at tray size;
  design monochrome-first with the status colour as tint.
