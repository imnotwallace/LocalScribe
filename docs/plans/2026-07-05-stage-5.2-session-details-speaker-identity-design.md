# Stage 5.2 Design: Session Details Window + Speaker & Identity Management

Date: 2026-07-05
Status: Approved-in-brainstorm (2026-07-05), grounded in two code-survey workflows
(wf `understand-stage-5.2`, wf `understand-speaker-identity`). Refines Stage 4
(manager) and Stage 5 (diarisation). Follows the smoke-driven UI checkpoint
(commit 92c756e) that added the icon-rail nav, full-width Sessions list, row
selection highlight, and an INTERIM Sessions details drawer (superseded here).

## 1. Scope

Stage 5.2 moves session metadata editing out of the cramped inline pane into a
dedicated window, and turns the participant/speaker model into a first-class
two-column speaker & identity manager that drives (and is driven by) diarisation.

Delivered:

- **W — Session Details window.** A dedicated, reusable window hosting the metadata
  editor, opened by session id from BOTH the Sessions grid ("Open detail") and the
  Matters page (tagged-session "Open"). Removes the interim Sessions drawer.
- **B1 — Two-column speaker & identity management.** Participants presented as two
  ordered LOCAL / REMOTE speaker lists; per-side counts derive from the lists (and
  still drive diarisation) with a manual override; roster-add gains a local/remote
  choice (fixing the "everything is remote" bug); a new session defaults to Local
  Speaker 1 = the app-settings self identity.
- **B2 — Identity ↔ cluster binding.** Wire the reserved `SessionParticipant.ClusterKey`
  so a named person binds to a diarised speaker slot ("all Remote Speaker 2 =
  Barrister"); a participant picker in the Split-speakers dialog replaces free-text-only
  naming; `NameResolver` honors the binding. Hot-swapping the person in a slot needs no
  re-diarise and touches no transcript lines.

### 1.1 Explicitly deferred (design-noted, NOT forgotten)

- **B3 — Line / time-range re-allocation** ("the first 30s were Remote-Speaker-2, not
  -3"). Its own future stage. See §7 for the carried design note — the non-destructive
  primitive already exists; what's deferred is the transcript-editing product surface.
- **Manager IA overhaul** (Sessions action bar; Matters paginated/sortable/searchable +
  Date-created/Date-archived columns + details drawer; untag-with-warning). Its own next
  stage; fully mapped already in the Stage-5.2 IA survey brief. The **"Open detail"**
  entry point for the window ships here via the **context menu**; the action-bar button
  arrives with that stage.

### 1.2 User decisions (2026-07-05)

| Question | Decision |
|---|---|
| Build order | Speaker management on its window first: window shell → B1 → B2; Manager overhaul and B3 are later stages. |
| Line re-allocation (B3) | Deferred to its own stage, but carried as a design note now (§7). |
| Window vs read view | The Session Details window **complements** the existing read-only transcript window (`ReadViewWindow`) — two distinct windows, two dedup maps. |
| Window entry contract | Windows open **by session id** (symmetric with `ReadViewWindow`), not by passing a live row object. |
| Identity vs side | Wire the reserved `ClusterKey` + an editable per-session slot↔identity mapping. Keep `SessionParticipant.Side`; do NOT do nullable-Side / slot-owns-side surgery now. |
| Counts vs lists | Per-side counts default-derive from the two lists but remain manually overridable (system-mix legs where declared ≠ real speakers). |

## 2. Current state (from the surveys)

- **Metadata editor** is `MetadataEditorViewModel` (WPF-free, `MetadataEditorViewModel.cs:34`),
  today an app-lifetime **singleton** built once (`App.xaml.cs:100-101`), handed to the one
  `SessionsPage`, attached to the current selection via `editor.Attach(vm.SelectedRow)`
  (`SessionsPage.xaml.cs:29-33`). It holds one `_row`, uses the row object as a
  `ConditionalWeakTable` key for the revert-base cache (`MetadataEditorViewModel.cs:57,136-138`)
  and for live-lock checks `_row.Id == CurrentSessionId` (lines 346-349). **One editor
  backs exactly one attached row** — it cannot back multiple windows as-is.
- **Windows** (`ReadViewWindow`, `SplitSpeakersWindow`) are minted fresh per open with
  their own VM, registered in `WindowRegistry` (close-all-for-delete), capture-excluded
  (`CaptureExclusion`), window-state-persisted (`WindowStateStore`), and (ReadView)
  deduped per session id via an App-level dictionary (`App.xaml.cs:132`).
- **Participants**: `SessionParticipant` (`SessionParticipant.cs:6-14`) = `Id, Name,
  Side (SourceKind Local/Remote, required), Role?, IsSelf, ClusterKey?` — where
  `ClusterKey` is reserved and **dead** (never set/read). The **self** participant is
  minted local at session start (`SessionBootstrap.cs:22-24`).
- **Roster** (`RosterMember`, `Matter.cs:4-9`) = `Id, Name, Role?` — **no side**; side is
  chosen per-session when copied in. Free-text add has a side checkbox
  (`NewParticipantIsRemote`, default `true`); **roster-add hard-codes Remote**
  (`MetadataEditorViewModel.cs:180`) — the "everything is remote" bug.
- **Counts**: `SessionMeta.LocalCount`/`RemoteCount` (`SessionMeta.cs:15-16`, default 1/1),
  hand-typed, **orthogonal to the participants list**. They already drive diarisation:
  a side's count → `DiarisationRequest.ForcedClusterCount` → sherpa `Clustering.NumClusters`
  (`SplitSpeakersViewModel.cs:183-189,272`, `SherpaDiarisationRunner.cs:23-24`).
- **Diarisation names**: `Speakers` overlay (`Speakers.cs:6-18`) = `Names` (clusterKey→name),
  `Assignments` (source→seq→clusterKey), `Pinned`. clusterKey = `"<Source>:<clusterId>"`.
  Cluster names are **free text** typed in the Split dialog (`SplitSpeakersWindow.xaml:84-85`),
  defaulting to `"{Source} Speaker N"`; no roster/participant picker exists.
- **Line attribution** is resolved at render by `NameResolver.Resolve` (`NameResolver.cs:9-36`),
  3-tier: (1) `Assignments[side][seq]` → `Names[clusterKey]`; (2) if that side's count == 1,
  the single participant's name; (3) baseline `SpeakerLabel`. Note tier (2) keys off
  `count == 1` — any list↔count coupling must preserve this.

## 3. W — Session Details window

### 3.1 Window
- New `SessionDetailsWindow` (`ui:FluentWindow`, Mica), modeled verbatim on
  `ReadViewWindow`: `ui:TitleBar` + Icon; `WindowRegistry.Register(sessionId, Close)` in
  ctor and `Unregister` in `OnClosed`; `CaptureExclusion.Apply` in `OnSourceInitialized`
  + re-apply on `ISettingsService.Changed` (guarded by `_hwndReady`); own `DispatcherTimer`
  → `editor.Tick()` started on show, stopped in `OnClosed`; `WindowStateStore` key
  `"sessionDetailsDefault"`. Safe as a FluentWindow because it opens on demand, long after
  the message pump is up (the invisible-Mica gotcha only bites pre-pump windows — memory
  `wpfui-fluentwindow-startup-rendering`).
- Content: the existing grouped editor cards (Session / Classification / Participants /
  Speakers) lifted from the interim drawer, PLUS the B1 two-column speaker manager (§4).

### 3.2 Per-window editor (retire the singleton)
- `MetadataEditorViewModel` becomes a **per-window factory output** — one fresh instance per
  window, exactly like `ReadViewViewModel`/`SplitSpeakersViewModel` (`App.xaml.cs:121-148`).
  All five ctor deps are ambient singletons (`MaintenanceService, SessionViewModel,
  IUiErrorReporter, Action<Action> dispatch, TimeProvider`, `MetadataEditorViewModel.cs:94-98`).
  The VM is WPF-free, so it moves cleanly.

### 3.3 Load by id (the one structural refactor)
- Open events are session-id strings; `Attach` needs a full `SessionRowViewModel`. Add an
  **id-first load** so both entry points work without a live row: a single-session load in
  `MaintenanceService` (reads that session's `session.json`+`meta.json`) producing the row
  the editor needs, exposed as `MetadataEditorViewModel.LoadAsync(string sessionId)` that
  builds/attaches internally. This preserves the editor's row-identity model (CWT keys,
  live-lock checks at `MetadataEditorViewModel.cs:346-349`) by constructing the row it keys
  on. This is the highest-risk change in the stage and gets its own task + review.

### 3.4 Wiring (App.xaml.cs)
- New `openSessionDetails` `Action<string>` (dedup dict `sessionDetailsWindows`, activate-or-
  create + `Closed` hook to evict + dispose; mirror `readViews`, `App.xaml.cs:127-142`),
  minting a fresh editor per window.
- Wire into `sessionsVm.OpenSessionDetailsRequested` (new event) and add an **"Open detail"**
  context-menu item to the Sessions grid. Wire into `mattersVm.OpenSessionDetailsRequested`
  and change the Matters tagged-session "Open" from `openReadView` to `openSessionDetails`.
- The read view stays reachable (double-click / "Open" on Sessions) — the two windows
  complement each other.

### 3.5 Remove the interim drawer
- After the window is wired from both pages, remove the Sessions inline `DetailsDrawer` +
  the singleton editor binding + `_editorTick` (`SessionsPage.xaml`/`.cs`). Sessions becomes
  a full-width list; details open in the window. Ordering invariant: never remove the drawer
  before the window works (there is always a way to view details).

## 4. B1 — Two-column speaker & identity management

Hosted in the Session Details window's editor. Replaces the flat participants list + the
two hand-typed count boxes with a two-column manager:

```
  Speaker management
  +-----------------------+-----------------------+
  | Local  (1 speaker)    | Remote  (3 speakers)  |
  +-----------------------+-----------------------+
  | 1  Samuel  (you)      | 1  Colleague          |
  |                       | 2  Barrister          |
  | + Add local           | 3  Client             |
  |                       | + Add remote          |
  +-----------------------+-----------------------+
  [x] Speaker counts follow these lists   (Local 1 / Remote 3)
      [ override ]  Local [ 1 ]  Remote [ 3 ]
```

- **Two ordered lists** built from `SessionMeta.Participants` split by `Side`. Position in a
  side = the speaker slot number ("Local Speaker 1"). Each row shows slot number + name
  (+ role, + "you" for `IsSelf`).
- **Add per side** with the side pre-chosen by which column's "Add" is used (roster picker or
  free text). Fixes Gap B by stamping the chosen `SourceKind` — no more hard-coded Remote
  (`MetadataEditorViewModel.cs:180`); the roster-add path gains the side.
- **Counts derive from the lists by default**: `LocalCount = #local`, `RemoteCount = #remote`,
  still persisted as the two ints (so diarisation is unchanged downstream). A **"follow lists"**
  toggle (default on) with a manual override for system-mix legs where declared ≠ real
  speakers (preserve the `SplitSpeakersViewModel.SystemMixWarning` escape hatch,
  `SplitSpeakersViewModel.cs:319-321`). Preserve the `NameResolver` tier-2 `count==1` behavior.
- **Self-default**: a new session starts with Local Speaker 1 = the app-settings self identity
  (`Settings` `SelfName`/`SelfRole`). Ensure the self participant minted at
  `SessionBootstrap.cs:22-24` takes its name/role from settings (verify + wire).
- **Identity vs slot**: the list position IS the slot; the name/role is the identity attached
  to it. Editable per-session (rename/reorder/move side). Keep `Side` on the participant record
  (no model rewrite).

Data-model impact: **App-only** (VM derives counts, still writes the two ints; side-on-add
stamps an existing field). No Core schema change for B1.

## 5. B2 — Identity ↔ cluster binding

- **Wire `SessionParticipant.ClusterKey`** (already declared, `SessionParticipant.cs:13`) as
  the person→diarised-slot link. Setting it is additive; old files keep `null`.
- **Participant picker in the Split dialog**: replace free-text-only cluster naming
  (`SplitSpeakersWindow.xaml:84-85`) with a picker over the session's per-side participants,
  so a cluster binds to a named person; free text remains available for un-rostered speakers.
- **`NameResolver` honors the binding**: before the existing tiers, if a participant's
  `ClusterKey` matches the seq's assigned cluster, resolve to that participant's name
  (`NameResolver.cs:9-36`). Keeps the display-overlay nature (no transcript mutation).
- Result: "all Remote Speaker 2 = Barrister"; swapping the person in a slot re-points the
  binding with no re-diarise and no line edits. Pins still survive re-diarise via
  `SpeakersMerge` collision-avoidance remap (Stage 5 guarantee).

Data-model impact: **additive Core** (start using the reserved `ClusterKey`; one new
`NameResolver` branch). Backward-compatible.

### 5.1 Link to the transcript viewer (read view)

The speaker/identity manager and the transcript viewer are **not** wired window-to-window;
they link through the **shared session data** the transcript resolves names from. There are
two directions:

- **Forward (ships this stage).** Identities and `ClusterKey` bindings set in the Session
  Details window persist to `meta.Participants` / `speakers.json`, and the read view resolves
  each line's speaker at render via `NameResolver` (`NameResolver.cs:9-36`). So binding
  "Remote Speaker 2 → Barrister", or naming the single local speaker "Samuel", makes every
  matching transcript line read that name — with **no re-diarise and no line mutation**. This
  is the payoff of B2 + B1 tier-2 (`count == 1`).
  - **Live refresh:** if a read view for the same session is already open when identities
    change, it must re-resolve to show the new names. Add a lightweight refresh path: on the
    editor's save-commit, signal any open `ReadViewWindow` for that session id (a small
    App-level notification keyed by session id, reusing the dedup maps) to re-run its
    projection/name resolution. Without this, the two windows silently disagree until the read
    view is reopened.
- **Reverse (B3, deferred — §7).** Editing attribution *from* the transcript — selecting a
  line/time-range and reassigning it to a different speaker — is what turns the viewer into an
  editor. That is the deferred stage; the read view stays read-only here.

**Convergence intent (user decision 2026-07-05: two complementary windows for now).** Keep
Session Details and the read view as separate windows this stage. For the future B3 editing
surface, separate **management** from **reassignment** so the speaker UI never overshadows the
transcript:
- **Management** (who the speakers are, sides, counts, identity↔cluster bindings) stays the
  heavy two-column grid in the **Session Details window** — set up once.
- **Reassignment in the transcript** only *picks from* the already-defined speakers, so it is a
  **lightweight** affordance, never the grid: select lines → a compact speaker picker (colored
  slot chips e.g. `L1 Samuel · R2 Barrister`, or a right-click "Reassign to…" list), plus at
  most a thin one-line speaker legend. The transcript remains the primary surface.
This keeps the door open to B3 without embedding the management grid into the reader.

## 6. Non-goals

- No `Side`-nullable / slot-owns-side participant rewrite (deferred; wire `ClusterKey` instead).
- No transcript line editing / re-allocation (B3, §7) — the read view stays read-only this stage.
- No Manager IA overhaul (action bar, Matters pagination/sort/search, date columns, drawer,
  untag) — next stage; only the "Open detail" context-menu entry point ships here.
- No new diarisation engine behavior; counts still feed `ForcedClusterCount` exactly as today.

## 7. Carried design note — B3 line / time-range re-allocation (future stage)

Recorded now so the next stage inherits it (do NOT drop):

- **The non-destructive primitive already exists and is tested**: `EditStore.ReassignSpeakerAsync`
  (`EditStore.cs:34-52`) pins a single `Assignments[source][seq] = clusterKey` + adds to
  `Pinned`, sets `meta.Edited`/`LastEditedAtUtc`; `SpeakersMerge` guarantees the pin survives
  any future re-diarise (incl. collision-avoidance remap). transcript.jsonl is never touched.
- **What's missing** for "first 30s were Remote-Speaker-2, not -3": (1) a thin **range→seqs**
  resolver (filter `TranscriptLine`s by side + `[StartMs,EndMs)`); (2) an optional **batched**
  `EditStore` wrapper (N seqs, one write, one edited-flag bump); (3) a UI affordance that
  reverses the read view's "Read-only by design" stance (`ReadViewWindow.xaml:12-13`) — a
  product decision, the "corrections" surface. Note `DisplayRow` drops seq granularity
  (`DisplayRow.cs:5-11`); selection UI must work from the pre-grouped `TranscriptLine`/segment
  list, not the rendered rows.
- **Evidentiary**: route the write through the `MaintenanceService` single-write gate (per-session
  semaphore + projection regen), NOT `EditStore`'s current direct write — reconcile that when
  wiring. Persist only as pinned per-seq overlays; never mutate transcript.jsonl. This matches
  the correction-only, non-destructive, no-delete invariant (memory
  `no-transcript-content-deletion-evidentiary`).

## 8. Testing strategy

TDD, subagent-driven, 0-warning gate, App.Tests (currently 153) + Core.Tests green (the 2
DER/golden fixture failures are pre-existing env-absence, not regressions).

- **VM/logic (real tests):** per-window editor `LoadAsync(id)` round-trip; counts-derive-from-lists
  incl. the manual-override and `count==1` `NameResolver` coupling; roster-add stamps the chosen
  side; self-default pulls Settings identity; `ClusterKey` binding round-trips and `NameResolver`
  honors it; participant picker sets `ClusterKey`.
- **Core additive:** if any Core change (self-default wiring, NameResolver branch), add store/
  resolver tests; keep changes backward-compatible (old files parse).
- **Static/UI guards:** existing `XamlHygieneTests` (no hardcoded ARGB) must stay green for all
  new XAML.
- **Manual GUI smoke (own runbook):** open details from Sessions + from a Matters tagged session;
  two-column lists + add per side + self-default visible; counts follow lists + override; bind a
  cluster to a participant in the Split dialog and see the name flow to the read view; drawer gone
  from Sessions; capture-exclusion + window-state + delete-closes-window all hold.

## 9. Constraints & risks

- **Load-by-id refactor (§3.3)** is the highest-risk change — it alters how the editor acquires
  its row. Isolate it in one task with focused review; keep the CWT/live-lock identity model
  intact by constructing the row.
- **Counts↔lists coupling** must not regress the system-mix override or `NameResolver` tier-2
  (`count==1`). Keep the manual override and a test for it.
- **Per-window editor** must not reintroduce cross-window state (each window its own VM + timer).
- **FluentWindow pre-pump gotcha**: the details window is opened on demand (safe); do not show it
  at startup.
- **Evidentiary**: B2 is display-overlay only (no transcript mutation); B3 stays out of scope.
