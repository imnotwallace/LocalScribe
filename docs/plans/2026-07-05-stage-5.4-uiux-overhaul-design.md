# Stage 5.4 — Session Details, Recording & Playback UX Overhaul — Design

- **Status:** Approved design (brainstorm complete). Awaiting spec review, then implementation plan.
- **Date:** 2026-07-05
- **Branch:** new `stage-5.4-uiux-overhaul` branched off the current
  `stage-5.3-sessions-matters-manager-ia`. Stage 5.4 edits the Session Details window and Manager IA
  that exist only in the unmerged 5.1/5.2/5.3 bundle, so it stacks on that branch; the eventual merge
  lands 5.1–5.4 together (same bundling already in flight). Branching from bare `master` is not
  viable.
- **Scope:** One combined spec, implemented in three phases.

---

## 1. Motivation

Live GUI use surfaced eleven UX defects/gaps spanning playback, recording discoverability, the
Session Details window, the Matters tab, transcript rendering, and app-wide theming. Rather than
patch them piecemeal, this is a single coordinated overhaul with a shared design, delivered in
phases so low-risk fixes can ship first.

The eleven concerns, and how they map to workstreams:

| # | Concern | Workstream / Phase |
|---|---------|--------------------|
| ① | Read-view playback bar frozen; nothing at end-of-media; drag-only seek; wants "serious overhaul" | A / Phase 1 |
| ② | No obvious way to start/stop/pause a recording; Record button wanted on the nav rail; Live window is a dead "Idle 00:00" | C / Phase 3 |
| ③ | Session Details auto-applies every edit on the fly; wants an explicit Save; destructive changes possible | B / Phase 2 |
| ④ | Sessions grid does not refresh after Session Details edits | A / Phase 1 |
| ⑤ | Speaker area (Participants lists vs "Split speakers" vs count boxes + "counts follow lists") is contradictory | B / Phase 2 |
| ⑥ | Matter assignment is a checkbox wall; wants a searchable multi-select ("what if I have 100 matters?") | B / Phase 2 |
| ⑦/⑩ | Accent buttons ("New matter", "Delete…", etc.) render dark/black text on a saturated fill — unreadable | A / Phase 1 |
| ⑧ | No way to untag a session from a matter's "Tagged sessions" list | B / Phase 2 |
| ⑪ | Transcript renders one speaker's whole turn as a single block; wants a same-speaker silence gap to start a new section, plus better spacing | A / Phase 1 |

(⑨ "Record button on LHS panel" folded into ②; ⑩ "Delete button contrast" folded into ⑦.)

## 2. Hard constraints & invariants (apply to every item)

These are load-bearing project rules confirmed during code mapping. Every change below must
preserve them.

1. **Evidentiary firewall.** `transcript.jsonl` and audio files are primary evidence and are
   **never** mutated for display, correction, or organizational purposes. Metadata edits touch
   only `meta.json`; diarisation/name changes are non-destructive overlays in `speakers.json`;
   `Edited`/`LastEditedAtUtc`/`Summary*` correction flags are set **only** by `EditStore` and must
   flow through `BuildMeta` untouched. No hide/delete/redact of transcript content. Audio retention
   default is "keep" (no auto-deletion).
2. **Single disk-write owner.** ViewModels never touch stores directly. All persistence goes through
   `MaintenanceService` under its per-session / index gate (`SaveMetaAsync`, `SetArchivedAsync`,
   `SaveDiarisationAsync`, `SaveMatterAsync`, `CreateMatterAsync`, `ListMattersAsync`,
   `LoadSessionItemAsync`, `ListSessionsAsync`).
3. **WPF-free ViewModels.** `PlaybackViewModel`, `ReadViewViewModel`, `MetadataEditorViewModel`,
   `SessionsPageViewModel`, `MattersPageViewModel`, `SessionViewModel`, etc. reference no WPF types.
   WPF (MediaPlayer, DispatcherTimer, MessageBox, windows) lives only in code-behind or behind test
   seams (`IDualAudioPlayer`). All background→UI mutation routes through the injected
   `Action<Action> _dispatch` (= `Dispatcher.BeginInvoke`); timers surface as a plain, directly
   testable `Tick()`.
4. **CommunityToolkit.Mvvm.** `[ObservableProperty]` + generated `OnXxxChanged` partials;
   `IRelayCommand`/`IAsyncRelayCommand` with real `CanExecute` gates + `NotifyCanExecuteChanged`.
5. **FluentWindow startup gotcha.** A Mica `FluentWindow` shown before the message pump renders
   invisible on this box. Any new top-level window must show after the pump (or use
   `WindowBackdropType=None`, as the Live/overlay windows already do). No change here creates a
   pre-pump window.
6. **Cross-VM communication = plain C# events** (`Action`/`Action<string>`) wired in the
   `App.xaml.cs` composition root (mirroring `OpenSessionDetailsRequested` / `DiariseRequested`).
   No messenger/event-aggregator is introduced.
7. **Permanently-dropped features (do not re-introduce):** meeting auto-detection as a primary
   start path, and global hotkeys (Webex Ctrl+Alt conflicts). Recording is manual-start.
8. **Test seams:** humble objects over the OS (`DualMediaPlayer`, windows, tray) are not
   unit-tested and are verified via the manual smoke runbook; all logic they wrap lives in
   WPF-free VMs that are unit-tested. The XAML hygiene test (`XamlHygieneTests`) forbids hardcoded
   ARGB and asserts theme conventions.

## 3. Locked design decisions (from brainstorm)

- Structure: **one combined Stage 5.4 spec, three phases.**
- ⑤ Speaker model: **identity-first with unnamed slots** — remove the integer count boxes and the
  "counts follow lists" checkbox; each side is a list of speaker slots (named + explicit unnamed);
  **wire `ClusterKey` now** so a named slot durably owns a detected voice.
- ③ Save model: **buffer everything** until Save; **Discard** reverts to last-saved; **prompt
  Save/Discard/Cancel on close** when dirty; **Save warns when speaker-label attribution changes.**
- ⑥ Matter picker: **search box + filtered list + removable chips**; **inline create** when no
  match; roll the searchable pattern out to the **Sessions-page matter filter** and the **Matters
  manager list** too.
- ⑧ Untag: add an **"Untag"** button in the matter's Tagged-sessions list, **with a confirm
  dialog.**
- ② Recording: **Record button on the left nav rail**; **Live window becomes a recording console**
  (Start when idle + capture summary + inline per-process app selector; live transcript + Pause/Stop
  when recording); **on Start, open the Live console + overlay pill**; **one-click start** (no
  pre-flight); **per-process app picker in Settings AND selectable at record time in the Live
  console.**
- ① Playback end-of-media: **hold at end, reset button to Play/Replay** (click restarts from 0);
  **all four upgrades** (click + keyboard seek, Stop/Restart, per-leg volume, transcript-synced
  highlight + click-to-jump).
- ⑦/⑩ Contrast: **global fix** (inheritable root foreground, remove the leaking implicit style).
- ⑪ Sectioning: **same-speaker gap ≥ ~5s starts a new section**, threshold **adjustable in
  Settings** (default 5000 ms); applies to **both live and read views**; display-only.

## 4. Phase 1 — Correctness & polish (low-risk, independently shippable)

### 4.1 Playback overhaul (①)

**Root causes (confirmed):**
- The seek slider and elapsed text advance *only* via a 150 ms `DispatcherTimer`
  (`ReadViewWindow.xaml.cs:42,65-70`). The timer's `Start()` is gated on `Playback.IsAvailable`
  (`ReadViewWindow.xaml.cs:63`), a flag published one dispatcher turn later inside `Apply()` via
  `_dispatch` = `Dispatcher.BeginInvoke` (`ReadViewViewModel.cs:115`, `App.xaml.cs:75`). The guard
  races the dispatched publish, so the timer often never starts → frozen 00:00.
- The `Slider` has **no** `Value`/`Maximum` binding (`ReadViewWindow.xaml:64-65`); it is poked only
  by the timer. The Play/Pause caption is set imperatively only in the click handler
  (`ReadViewWindow.xaml.cs:100-104`), never bound to `IsPlaying`.
- `MediaEnded` flips `IsPlaying=false` but does not reset the caption or rewind; next Play resumes
  from the end position with no seek-to-0, so replay is silent
  (`PlaybackViewModel.cs:45-49`).
- Seek is thumb-drag only (`Thumb.DragCompleted` → `Playback.Seek`); track-click and keyboard do
  nothing (`ReadViewWindow.xaml.cs:108-114`).

**Design:**
- **Deterministic timer start + bindings.** Bind the slider `Value`/`Maximum` and the Play/Pause
  caption to VM state. Add `[ObservableProperty]` `IsPlaying`, `PositionMs`, `DurationMs` (already
  present in part). Start the position timer from where `IsAvailable` is actually published (inside
  `Apply` on the same dispatcher turn, or by reacting to an `IsAvailable` change) rather than the
  post-`await` guard. Keep a directly-callable `Tick()` for tests. The slider/caption no longer
  depend solely on the timer firing.
- **End of playback = hold + reset to Play.** `MediaEnded` → `IsPlaying=false`, caption → "Play",
  position **held at end** (no auto-rewind). Track an `EndReached` state so the next Play seeks to 0
  first (fixes replay).
- **Click + keyboard seek.** Enable move-to-point on the slider and route track-click / arrow keys
  through `Seek`; suppress the timer's snap-back while the user is interacting (extend the existing
  `_seekDragging` guard to a general "user scrubbing" flag).
- **Stop/Restart button.** New transport button → Pause + `SeekMs(0)` + caption "Play".
- **Per-leg volume.** Extend `IDualAudioPlayer` with `SetLegVolume(bool local, double volume)`;
  `DualMediaPlayer` sets `MediaPlayer.Volume` per leg. Add `LocalVolume`/`RemoteVolume`
  (0.0–1.0) to `PlaybackViewModel` with volume sliders in the transport. Keep the existing per-leg
  mute toggles.
- **Transcript-synced highlight + click-to-jump.** The read view's `Rows` are `DisplayRow`s
  carrying `StartMs`. During `Tick()`, compute the "now playing" section = the row whose
  `[StartMs, nextRow.StartMs)` window contains `PositionMs`; expose it as a `PlayingIndex` the
  `ListView` highlights. Clicking a section calls `Seek(row.StartMs)` + Play. (Depends on §4.2's
  `StartMs`/section model, hence Phase 1 pairing.)

**Files:** `ReadViewWindow.xaml`, `ReadViewWindow.xaml.cs`, `ViewModels/PlaybackViewModel.cs`,
`ViewModels/ReadViewViewModel.cs`, `DualMediaPlayer.cs`, `Services/IDualAudioPlayer.cs`.

**Testing:** extend `PlaybackViewModelTests` + the `FakePlayer` for volume, end-reached replay,
and the now-playing highlight index. `DualMediaPlayer` stays humble (smoke-verified).

### 4.2 Transcript section grouping + spacing (⑪)

**Root cause (confirmed):** `TranscriptProjection.Build` groups consecutive segments purely by
`last.DisplayName == p.Name` with **no time logic** (`TranscriptProjection.cs:52-64`), so a
speaker's whole turn merges into one paragraph regardless of pauses. VAD is *not* the problem —
each `TranscriptLine` is a separate VAD segment carrying `StartMs` **and** `EndMs`
(`TranscriptLine.cs:11-12`); the display simply discards the gaps. The live view
(`TranscriptLinesViewModel.cs:22-39`) does *no* grouping (one row per segment), so live and read
already diverge (acknowledged at `ReadViewViewModel.cs:17-19`).

**Design (display-only, non-destructive — `transcript.jsonl` untouched):**
- Extract a pure **`SectionGrouper.Group(preRows, gapMs)`** helper. Merge predicate becomes
  `same DisplayName AND (p.StartMs − runningSectionEndMs) < gapMs`; otherwise start a new section.
- Add `EndMs` to `PreRow` (`TranscriptProjection.cs:13`) and to `DisplayRow` (`DisplayRow.cs`), so a
  merged section carries the running end of its last segment for the next gap comparison.
- Thread the threshold from `Settings.SectionGapMs` (**default 5000 ms**, adjustable in Settings)
  into `TranscriptProjection.Build` (callers `SessionWriter.cs:51` and `ReadViewViewModel.cs:105`
  both have `Settings` in hand).
- **Live/read consistency:** have the live view run the *same* `SectionGrouper` over the merger's
  full sorted `View` on each `LineInserted` (expose `merger.View` via `SessionController`), so a
  late out-of-order insert re-splits correctly. Minimum for this feature: identical section
  *boundaries* in both views.
- **Better spacing:** add an inter-section `Margin` on the item container in both
  `ReadViewWindow.xaml:81-85` and `LiveViewWindow.xaml:28-32`, and restructure the item template so
  the speaker header sits on its own line above the paragraph. Add a blank line between sections in
  `PlainTextRenderer`/`MarkdownRenderer` for the exported `.txt`/`.md`.

**Files:** `Core/Projection/TranscriptProjection.cs`, `Core/Projection/DisplayRow.cs`,
new `Core/Projection/SectionGrouper.cs`, `Core/Model/Settings.cs` (add `SectionGapMs`),
`Core/Live/SessionController.cs` (expose `View`), `ViewModels/TranscriptLinesViewModel.cs`,
`ReadViewWindow.xaml`, `LiveViewWindow.xaml`, `Core/Projection/PlainTextRenderer.cs`,
`Core/Projection/MarkdownRenderer.cs`, `SettingsPage.xaml` (threshold control).

**Testing:** `SectionGrouper` unit tests (gap boundary at/above/below threshold, speaker change,
marker handling, out-of-order insert); projection tests for the new `EndMs` plumbing.

### 4.3 Accent-button contrast — global fix (⑦/⑩)

**Root cause (confirmed):** the app-global **implicit `TextBlock` style** in `Fluent.Shared.xaml:8-10`
(`Foreground = TextFillColorPrimaryBrush`) — added so loose TextBlocks are visible on Mica — also
applies to the `TextBlock` a button's `ContentPresenter` auto-generates for string content. As a
Style-Setter it *overrides* the accent-button template's intended `TextOnAccentFillColorPrimaryBrush`
(white), painting near-black text on Primary/Danger fills in light theme. Affects **all** accent
buttons (New matter, Delete…, Add member, Session Details Adds). The button XAML itself is correct
(no ARGB, identical to peers).

**Design (Option A — global):** stop using an implicit `TextBlock` *Style* for default text color.
Instead set the **inheritable** `TextElement.Foreground` (`= TextFillColorPrimaryBrush`) on each
Page/Window **root** container. Inherited values do not override a value set closer in the tree, so
accent-button templates keep their white `TextOnAccent` foreground while loose TextBlocks still
inherit a visible color on Mica. Update `XamlHygieneTests` (`:26-34`) which currently asserts the
implicit style exists.

**Files:** `Styles/Fluent.Shared.xaml`, each page/window root
(`MainWindow.xaml`, `Pages/*.xaml`, `ReadViewWindow.xaml`, `LiveViewWindow.xaml`,
`SessionDetailsWindow.xaml`, `SplitSpeakersWindow.xaml`, `OverlayWindow.xaml`, `SettingsPage.xaml`),
`tests/.../XamlHygieneTests.cs`.

**Testing:** update hygiene test; manual visual verify of Primary + Danger buttons in light **and**
dark themes.

### 4.4 Sessions grid auto-refresh (④)

**Root cause (confirmed):** the editor and the grid hold **separate immutable** `SessionRowViewModel`
instances loaded from disk at different times; the grid never learns of a save. The editor raises
only `DiariseRequested` — no "saved" event (`MetadataEditorViewModel.cs:120`). Rows are immutable
(`SessionRowViewModel.cs:16`, no `INotifyPropertyChanged`), so a refresh must **replace** the row.
No messenger exists.

**Design:**
- Add an `event Action<string>? Saved` to `MetadataEditorViewModel`, raised on a successful commit
  (the `PersistAsync` success continuation, `:407-427`, gated by the settled `seq == _saveSeq`
  check at `:424`). In Phase 1 this fires on auto-save-settle; in Phase 2 it fires on the explicit
  Save commit — the grid-side wiring is identical either way.
- Add `SessionsPageViewModel.RefreshRowAsync(sessionId)`: reload just that session via
  `MaintenanceService.LoadSessionItemAsync`, build a fresh `SessionRowViewModel`, swap it in `_all`
  and `Rows` (ObservableCollection indexer → Replace), then re-run `RebuildMatterOptions()` +
  `ApplyFilters()` (which already re-selects by id, preserving selection). A full `LoadAsync` is the
  fallback.
- Wire in `App.xaml.cs`: the `openSessionDetails` factory subscribes `detailEditor.Saved +=
  sessionId => sessionsVm.RefreshRowAsync(sessionId)` (mirroring the `DiariseRequested` wiring at
  `:167`); also refresh on window `Closed` as a backstop. Refresh regardless of whether the details
  window was opened from the Sessions page or the Matters page jump.

**Files:** `ViewModels/MetadataEditorViewModel.cs`, `ViewModels/SessionsPageViewModel.cs`,
`App.xaml.cs`.

**Testing:** `SessionsPageViewModel` targeted-refresh unit test (row swapped, selection preserved,
matter options rebuilt).

## 5. Phase 2 — Session Details redesign

### 5.1 Explicit Save / Discard model (③)

**Root cause (confirmed):** every `[ObservableProperty]` setter's `OnXxxChanged` calls `QueueSave()`
(`:340-347`), and collection/tag edits call it too — immediate write-through to `meta.json` with
only a 2 s "Saved" toast, no dirty tracking, no confirm, no `OnClosing` hook (a title typed then
closed via X can be lost). The buffer/commit/discard machinery is ~90% present: the VM edits an
in-memory working copy; `_savedMeta` is the last-saved baseline; `BuildMeta()` (`:384-396`) is the
commit projection; private `RevertToSaved()` (`:439-445`) is a discard.

**Design:**
- **Buffer everything.** `OnXxxChanged` partials and collection/tag edits set `IsDirty = true`
  instead of calling `QueueSave()`. Nothing persists until Save.
- **Save command** (`IAsyncRelayCommand`, `CanExecute = IsDirty && IsEditable`): commit via the
  existing `BuildMeta()` → `MaintenanceService.SaveMetaAsync` exactly once, computing the matter-tag
  delta against `_lastSavedTags` at commit time; update `_savedMeta`/`_lastSavedTags`; clear
  `IsDirty`; raise `Saved`.
- **Discard command** (`CanExecute = IsDirty`): promote `RevertToSaved()` to a command — reload
  fields from `_savedMeta`; clear `IsDirty`.
- **Close confirm.** Add `OnClosing` in the window code-behind: if `VM.IsDirty`, prompt
  **Save / Discard / Cancel** (Save awaits commit; Cancel sets `e.Cancel`). Fixes the lost-edit bug.
- **Attribution warning.** On Save, if the committed participant set changes rendered attribution
  (a named slot removed/renamed, or a side's single-named→count==1 labeling changes), show a
  one-line confirmation naming the affected labels before writing. Transcript content is never
  touched — only projected labels.
- **Indicator + shortcut.** Replace the transient toast with a persistent "Unsaved changes"
  indicator + enabled Save (greyed when clean). Ctrl+S = Save. Save/Discard disabled under the
  existing `IsEditable`/`LockHint` gate (live-recording / pending-recovery rows).
- **Scope note:** only the **Session Details** window converts to explicit Save. The **Matters-tab**
  details panel stays auto-save (its edits are organizational; destructive-ish actions get confirms
  — see §5.4).

**Files:** `ViewModels/MetadataEditorViewModel.cs`, `SessionDetailsWindow.xaml`,
`SessionDetailsWindow.xaml.cs`. (Precedent: `SplitSpeakersWindow.xaml:53,70` Cancel + Confirm/Save.)

**Testing:** dirty tracking per field/collection; Save commits once with correct tag delta; Discard
restores baseline; close-when-dirty branches; attribution-change warning fires only when labels
change; evidentiary flags untouched by `BuildMeta`.

### 5.2 Identity-first speaker model + durable identity↔voice link (⑤)

**Root cause (confirmed):** the speaker card fuses **three** representations of "who/how many
speak": named participants (`meta.Participants`, the two lists), per-side integer voice counts
(`meta.LocalCount`/`RemoteCount`, the count boxes), and diarised clusters (`speakers.json`, from
Split-speakers). The integer count exists only because a system-mix leg can carry more voices than
named people; it also drives diarisation `ForcedClusterCount` (`SplitSpeakersViewModel.cs:195-200,
289-290`) and `NameResolver` tier-2 (count==1 → label all of a side's lines with its single named
participant, `NameResolver.cs:26-31`). The "counts follow lists" checkbox (`CountsFollowLists`) is a
fragile bridge that auto-sets OFF when declared ≠ named (`:212-214`). The reserved
`SessionParticipant.ClusterKey` (`:13`) is the intended-but-dead structural link between identity
and cluster.

**Design — collapse three concepts into one "speaker slots" list per side:**
- A **speaker slot** is a `SessionParticipant` carrying a new **`Kind` enum**
  (`ParticipantKind { Named, Unnamed }`, default `Named`). A **named** slot has a `Name` (+ optional
  roster provenance, `IsSelf` for the local "you"); an **unnamed** slot is stored explicitly with
  `Kind = Unnamed` and an empty `Name`, its own `Id`, required `Side`, and an optional `ClusterKey`,
  and renders "Speaker N". Storing unnamed slots explicitly (rather than as a derived residual) gives
  them stable identity and ordering and lets a later diarised cluster attach to a specific slot. The
  field is additive and round-trips losslessly.
- **Per-side voice count = slot count.** Remove the integer boxes, `CountsFollowLists`, and
  `CountsAreManual` from the UI entirely. The persisted `meta.LocalCount`/`RemoteCount` remain the
  **pipeline-facing** value and are set on commit (`BuildMeta`) = the side's slot count, so
  downstream `ForcedClusterCount` (diarisation) and `NameResolver` tier-2 keep working **unchanged**
  (a lone named slot ⇒ count==1 ⇒ that name labels the side; a named + unnamed mix ⇒ count>1 ⇒
  clusters / "Speaker N"). The "declared can exceed named" system-mix case is now expressed by
  explicit unnamed slots instead of a raw integer.
- **Migration (lazy, non-destructive).** Existing sessions store named participants + an integer
  count with no unnamed rows. When the editor loads such a session, synthesize `Unnamed` slots on
  each side up to its declared count (`count − namedCount`); they persist on the next Save. Non-editor
  consumers (`NameResolver`, diarisation) keep reading the integer count and never require the
  unnamed rows, so unmigrated sessions render and diarise correctly until they are next saved. No
  batch migration pass is needed.
- **Wire `ClusterKey` (durable identity↔voice ownership).** When Split-speakers produces clusters,
  a named slot may **own** a cluster: on Confirm, write `speakers.Names[clusterKey]` (as today) *and*
  set the owning slot's `ClusterKey`. `NameResolver` prefers a slot's name for the cluster it owns.
  On **re-diarise**, preserve slot↔cluster ownership using the existing pin-preserving
  collision-avoidance remap (`SpeakersMerge`), never mutating `transcript.jsonl` and keeping the
  round-trip lossless. This unifies the three representations into one identity model.
- **Independent roster pickers.** Give each side its own "Add from roster" selection (remove the
  shared `SelectedRosterPick` that made "which column applies" ambiguous).
- **UI (Speakers card):** two columns (Local / Remote), each an ordered list of slot rows (named
  show name, "(you)" for self; unnamed show "Speaker N"), each row removable; per side: "＋ Add
  speaker" (free text), "＋ Add unnamed speaker", and an independent "Add from roster" picker.
  "Split speakers…" stays and uses the slot count as its target.

**Evidentiary care (adversarial-review focus):** manual pins survive re-diarise; no transcript
mutation; `ClusterKey` round-trips; removing/renaming a slot only changes projected labels (with the
§5.1 Save warning). This item touches Stage-5 diarisation code and will be built with TDD + a
whole-branch adversarial review, per the project's established pattern.

**Files:** `SessionDetailsWindow.xaml`, `ViewModels/MetadataEditorViewModel.cs`,
`Core/Model/SessionParticipant.cs` (add `ParticipantKind` enum; `ClusterKey` now live),
`Core/Model/SessionMeta.cs` (counts derived on commit), `Core/Projection/NameResolver.cs`
(prefer owned-cluster slot name), `ViewModels/SplitSpeakersViewModel.cs` +
`SplitSpeakersWindow.xaml` (seed from slots; set `ClusterKey` on confirm),
`Core/Diarisation/SpeakersMerge.cs` (preserve ownership on re-diarise),
`Core/Storage/SessionBootstrap.cs` (self slot).

**Testing:** slot model (named/unnamed/self round-trip); count-derivation on commit;
`NameResolver` for lone-named, named+unnamed, and owned-cluster cases; `SpeakersMerge` ownership
durability across re-diarise; migration of existing sessions (declared>named opens as named +
unnamed slots).

### 5.3 Searchable matter picker + inline create (⑥)

**Root cause (confirmed):** the relationship is already many-to-many (`meta.MatterIds` is an array;
no join table), so this is a pure widget swap. The current checkbox `ItemsControl`
(`SessionDetailsWindow.xaml:58-67`) materializes one checkbox per matter — unusable at ~100.
`_selectedMatterIds` (`:48`) is the selection truth (survives filtering); `TaggedMatters` (`:89`) is
already populated for a chips row but unshown; `RebuildMatterOptions` (`:498-511`) is the rebuild
hook.

**Design (Session Details):**
- Replace the checkbox wall with: a `ui:TextBox` **search box** (`MatterSearchText` +
  `OnMatterSearchTextChanged` → `RebuildMatterOptions`, mirroring `OnShowArchivedMatters`), matching
  **Name + Reference + Id** (case-insensitive, the app's `Contains(OrdinalIgnoreCase)` idiom); a
  short results list to tag; and a **removable-chips** row bound to `TaggedMatters` (x → the
  existing tag-toggle, buffered under §5.1 Save). Archived matters appear in results with an
  "(archived)" suffix.
- **Inline create:** when the search matches nothing, offer `Create matter "{text}"` →
  `MaintenanceService.CreateMatterAsync` (mints a per-day id) then tags it. Note: creating the matter
  writes immediately (additive/non-destructive); the session↔matter **tag** is buffered until Save.
- Selection continues to route only through `_selectedMatterIds` + the tag-toggle path so
  `SaveMetaAsync`'s incremental `SessionCount` delta stays correct.

**Roll-out (same pattern):**
- **Sessions-page matter filter:** replace the single-select combo with the searchable control.
- **Matters manager list:** add a search box filtering the left-hand matter list.

**Files:** `SessionDetailsWindow.xaml`, `ViewModels/MetadataEditorViewModel.cs`,
`Pages/SessionsPage.xaml` + `ViewModels/SessionsPageViewModel.cs`,
`Pages/MattersPage.xaml` + `ViewModels/MattersPageViewModel.cs`. Reuse the existing Chip style
(`SessionsPage.xaml:14-19`).

**Testing:** filter matches Name/Reference/Id; selection survives filtering out a selected matter;
archived surfacing; inline-create mints + tags; `SessionCount` delta correctness.

### 5.4 Untag session from a matter (⑧)

**Root cause (confirmed):** the Tagged-sessions rows have only an "Open" button
(`MattersPage.xaml:96-112`); no untag command exists on the Matters side, even though the full write
path does (`SaveMetaAsync` computes the tag delta and decrements `SessionCount`). Delete-matter
refuses while any session is tagged (`MattersPageViewModel.cs:262-267`, `MatterDeleter.cs:20-23`),
so untag is the missing step to correct a mistag or reach zero references.

**Design:**
- Add an **"Untag"** button beside "Open" in each tagged-session row (`Tag="{Binding SessionId}"`,
  mirroring the roster Remove pattern), gated behind a **confirm dialog**.
- New `MattersPageViewModel.UntagSessionAsync(sessionId)`: `LoadSessionItemAsync` → build
  `meta with { MatterIds = previous − matterId }` → `SaveMetaAsync(sessionId, updated, previous)`
  (reads `previous` fresh for a correct −1 delta) → `RefreshAsync()` + `SelectAsync(matterId)` to
  update the card's "N session(s)" count and drop the row. Idempotent (no-op if not tagged).
- **Cross-window staleness — block while open.** If a Session Details window is open on that session
  (detected via `WindowRegistry`, which tracks open per-session windows by id), **disable Untag**
  with a "close the open Session Details window for this session first" note. This is the safe,
  predictable choice: it avoids clobbering unsaved edits in the open editor and cannot corrupt the
  matter's `SessionCount` delta. The scenario is rare.

**Files:** `Pages/MattersPage.xaml`, `Pages/MattersPage.xaml.cs`,
`ViewModels/MattersPageViewModel.cs`. Non-destructive/reversible (organizational only).

**Testing:** untag delta + refresh; idempotency; stale-editor guard.

## 6. Phase 3 — Recording lifecycle & discoverability (②)

**Root cause (confirmed):** the entire lifecycle is bound to the **tray right-click menu**
(`TrayIconHost.cs:54-56`); none of the three windows the user looks at has a Start control. The main
shell's `NavigationView` has only Sessions/Matters/Settings (`MainWindow.xaml:33`); the Sessions
empty-state literally says "press Record in the tray". The Live window is a passive viewer over the
shared `SessionViewModel`, so it shows "Idle 00:00" and opening it arms nothing
(`LiveViewWindow.xaml:14-22`). Capture source is silently resolved from Settings at Start; per-process
mode has no GUI to name the app.

**Design (all bind the existing shared `SessionViewModel` + its `CanExecute` gates — no new
lifecycle logic):**
- **Nav-rail Record.** Add a prominent Record entry to the shell nav rail bound to
  `SessionViewModel.StartCommand` (hand the shared VM to the shell/nav VM as it is already handed to
  `SessionsPageViewModel`/`OverlayViewModel`). The shell reflects Recording/Paused state.
- **Live window → recording console** (state-driven layout):
  - *Idle:* a prominent **Start/Record** button + a summary of what will be captured (resolved
    remote source + mic) + an **inline per-process target-app selector** (bound to `Settings.Remote.App`
    when `Mode = PerProcess`, editable before Start).
  - *Recording/Paused:* the live transcript + Pause/Stop (existing).
- **On Start:** open/activate the Live console **and** show the overlay pill.
- **One-click start:** keep `StartAsync` resolving source from Settings via `RemoteCapturePlanner`
  (per-process Webex/Zoom, system-mix Teams/browser); keep the after-the-fact balloon warnings. No
  blocking pre-flight modal.
- **Per-process app picker:** add an app picker to `SettingsPage` for `PerProcess` mode — this is
  the **persistent default** (`Settings.Remote.App`). Expose the same selector inline in the Live
  console idle state as a **per-session override**: it seeds from the Settings default and changes
  the target for *this* recording only; it does **not** write back to Settings, so the next session
  reverts to the saved default. Settings stays the single source of truth.
- **Shell recording status:** the main window shows a Recording/Paused/Idle + elapsed status
  indicator bound to the shared VM (so state is visible without the tray).
- Tray remains the canonical consent/status surface; window controls are equal peers. Exit still
  confirms + awaits `StopAsync`. No auto-start, no hotkeys.

**Files:** `MainWindow.xaml` (+ shell/nav VM), `LiveViewWindow.xaml` + `.xaml.cs`,
`ViewModels/SessionViewModel.cs` (surface capture summary / app override; no new lifecycle),
`SettingsPage.xaml` + settings VM (app picker), `Core/Model/Settings.cs` (`RemoteSetting.App`
already present), `App.xaml.cs` (open Live console + pill on Start), `Pages/SessionsPage.xaml`
(empty-state copy). Respect the FluentWindow startup gotcha and capture-exclusion for transcript
windows.

**Testing:** existing `SessionViewModel`/`SessionController` tests; console idle↔recording state
tests; settings app-picker persistence; the shared-VM binding across nav rail / console / overlay.
